You asked your AI agent a simple question: *"What was total revenue last quarter?"*

It wrote SQL. It ran. It returned a number. And that number was wrong.

Maybe it used `AVG` instead of `SUM`. Maybe it forgot to filter out cancelled orders. Maybe it joined on the wrong key and silently doubled every row. You didn't notice until someone put the number in a board deck and the CFO asked why it didn't match the finance report.

This is the most dangerous kind of bug — the kind that looks correct.

## The Problem Is Bigger Than You Think

AI coding assistants are remarkably good at SQL syntax. They rarely write queries that fail to execute. The problem is that they write queries that execute perfectly and return the wrong answer.

A 2024 study by Numbers Station found that LLMs achieve only 40-60% accuracy on real-world business queries, even when given the full schema. The queries parse. They run. They return plausible-looking numbers. But they're wrong in ways that are almost impossible to catch without deep domain knowledge.

Here are the most common failure modes:

**Wrong aggregation.** The agent uses `AVG(revenue)` when the metric is defined as `SUM(revenue)`. Both are valid SQL. Both return a number. Only one is right. Without knowing the business definition of "total revenue," the agent is guessing.

**Missing required filters.** Your `transactions` table includes cancelled, refunded, and test orders. Every revenue query must filter `WHERE status = 'completed'`. The agent doesn't know this. It queries the raw table and inflates the number by 15-30%.

**Unsafe joins.** The agent joins `orders` to `order_items` without understanding the grain. A many-to-many join through a bridge table silently duplicates rows, doubling or tripling aggregated metrics. The result looks right — it's just 3x too high.

**Stale or deprecated tables.** Your warehouse has `v1_customers`, `v2_customers`, and `customers_staging`. The agent picks one. It might pick the staging table with incomplete data, or the deprecated v1 table that hasn't been updated in six months.

**Ambiguous column names.** There are five columns named `status` across your schema. The agent picks the wrong one, or worse, joins two tables on `status` columns that have completely different value sets.

## Why This Happens

The root cause is simple: **AI agents can see your schema, but they can't see what it means.**

When an LLM connects to a database (via MCP, a SQL tool, or direct access), it sees table names, column names, and data types. That's it. It gets something like:

```
transactions (
  id INTEGER,
  revenue DECIMAL,
  status VARCHAR,
  created_at TIMESTAMP,
  customer_id INTEGER
)
```

From this, the agent knows there's a `revenue` column of type `DECIMAL`. It does not know:

- Whether revenue should be summed, averaged, or counted
- That `status = 'completed'` is a required filter for any revenue metric
- That `customer_id` joins to `customers.id` as many-to-one
- That this table replaced `legacy_transactions` six months ago
- That the `revenue` column is in cents, not dollars

The agent fills in these gaps with statistical guesses from its training data. Sometimes the guesses are right. Often they're not. And you won't know until the wrong number reaches a decision-maker.

## The Fix: Semantic Metadata via MCP

The solution is not a better prompt. It's not fine-tuning. It's not RAG over your documentation. The solution is giving your AI agent the **semantic context** it needs — structured metadata that describes what your data means, not just what it's named.

This is what a semantic plane does. It sits between your database and your AI tools, providing machine-readable answers to the questions that cause wrong SQL:

```yaml
# What aggregation to use
- column: revenue
  semantic_role: measure
  aggregation: SUM
  unit: USD (cents)

# What filters are required
- guardrail:
    column: status
    operator: "="
    value: "completed"
    reason: "Exclude cancelled, refunded, and test transactions"

# What the correct query looks like
- golden_query:
    question: "What is total revenue?"
    sql: |
      SELECT SUM(revenue) / 100.0 AS total_revenue_usd
      FROM transactions
      WHERE status = 'completed'
```

When the semantic plane is served via MCP (Model Context Protocol), the AI agent can look up this metadata before writing SQL. It doesn't guess at the aggregation — it reads it. It doesn't forget the filter — it's a guardrail. It doesn't need to pick between five `status` columns — the semantic layer tells it which one matters and why.

### What makes this different from a data dictionary?

A data dictionary is documentation for humans. A semantic plane is structured metadata for machines. The difference:

- **Data dictionaries** live in Notion, Confluence, or a wiki. AI agents can't reliably parse them.
- **Semantic planes** are structured YAML files served via MCP. AI agents read them programmatically.
- **Data dictionaries** describe what columns exist. **Semantic planes** describe how to use them — aggregation rules, guardrails, golden queries, join paths, and business context.

### What makes this different from a metrics layer like dbt Metrics?

A metrics layer defines pre-computed metrics. A semantic plane describes the raw data so AI agents can write any query correctly — including ad hoc queries that nobody pre-defined. They're complementary: if you have a metrics layer, the semantic plane references it. If you don't, the semantic plane fills the gap.
