Your most experienced data analyst knows things that are not written down anywhere. They know that the `revenue` column is in cents, not dollars. They know that `customer_type = 'internal'` must be filtered out of all customer counts. They know that the `orders` table was migrated in Q3 2024 and everything before that date should be pulled from `legacy_orders` instead.

This is institutional knowledge — the accumulated understanding of how your data actually works, built up over years of working with it. It lives in people's heads, in Slack threads, in meeting notes that nobody indexed. And it is exactly the knowledge that AI agents lack.

## Why AI Agents Make Domain Errors

When an AI coding assistant connects to your database, it sees structure. Table names, column names, data types, foreign keys. From this, it can generate syntactically correct SQL for almost any question. But syntax is not semantics.

The errors AI agents make are not random. They follow a predictable pattern — they are the same mistakes a new analyst would make on their first week. The agent does not know what a senior team member knows:

**Business rule errors.** The agent queries all transactions when only completed ones should count. It includes internal test accounts in customer metrics. It uses gross revenue when the company reports net. These are not schema errors — the SQL is valid. The results are wrong because the agent does not know the business rules.

**Historical context errors.** The agent queries a table that was deprecated six months ago. It uses a column that was repurposed after a migration. It does not know about the data quality issue in March that requires filtering out two weeks of records. These are things that only people who were there would know.

**Semantic ambiguity errors.** There are three columns called `status` in different tables, each with different value sets. The agent picks the wrong one, or joins on `status` assuming the values match. There are two tables that both look like they contain customer data — one is the source of truth, the other is a denormalized cache that is sometimes stale. The agent cannot tell the difference from the schema alone.

**Aggregation errors.** The agent averages a column that should be summed. It counts distinct when it should count all. It groups by month when the metric is defined as a rolling 28-day window. These are business definitions that exist in someone's head but not in any machine-readable format.

Every one of these errors has the same root cause: the AI agent is missing institutional knowledge that experienced team members take for granted.

## The Tribal Knowledge Problem

Institutional knowledge is not documentation. Documentation describes how systems are *supposed* to work. Institutional knowledge describes how they *actually* work — including the exceptions, the edge cases, the historical accidents, and the unwritten rules.

This knowledge is tribal. It is passed from person to person through onboarding sessions, code reviews, Slack messages, and the experience of making a mistake and being corrected. It is extraordinarily valuable and almost entirely invisible.

The problem compounds as teams grow and change. When the analyst who knows all the business rules leaves, that knowledge walks out the door. When a new team member joins, they spend weeks or months rediscovering rules that were never written down. And when an AI agent connects to the database, it starts with zero institutional knowledge and no way to acquire it.

This is not a failing of the AI. It is a failing of how we manage knowledge. We have spent decades building systems to store and query data, but almost no investment in storing and serving the *meaning* of that data in a machine-readable format.

## Encoding Institutional Knowledge as Metadata

The solution is to take the institutional knowledge that lives in people's heads and encode it as structured, machine-readable metadata. Not as a wiki page that gets outdated, not as a comment in a SQL file that nobody reads, but as versioned YAML that is served directly to AI tools at query time.

This is what ContextKit's semantic layer does. Each piece of institutional knowledge maps to a specific metadata structure:

### Business rules become guardrails

*"Always filter out cancelled orders when calculating revenue"* becomes:

```yaml
guardrails:
  - column: status
    operator: "="
    value: "completed"
    reason: "Exclude cancelled, refunded, and test orders from all revenue metrics"
```

### Metric definitions become semantic roles

*"Revenue should always be summed, never averaged"* becomes:

```yaml
- name: revenue
  semantic_role: measure
  aggregation: SUM
  description: "Transaction revenue in USD cents"
```

### Historical context becomes descriptions

*"This table replaced legacy_transactions in Q3 2024"* becomes:

```yaml
- name: transactions
  description: "Primary transaction table (replaced legacy_transactions in Q3 2024). Use this for all queries after 2024-07-01."
```

### Domain-specific meanings become annotations

*"Status 3 means cancelled, status 4 means refunded"* becomes:

```yaml
- name: status
  description: "Transaction status code: 1=pending, 2=completed, 3=cancelled, 4=refunded"
  semantic_role: dimension
```

### Known-correct answers become golden queries

*"Here's how to calculate quarterly revenue correctly"* becomes:

```yaml
golden_queries:
  - question: "What is total revenue this quarter?"
    sql: |
      SELECT SUM(revenue) / 100.0 AS revenue_usd
      FROM transactions
      WHERE status = 2
        AND created_at >= date_trunc('quarter', current_date)
```

Each of these transformations takes institutional knowledge from an implicit, fragile form (someone's memory) to an explicit, durable form (versioned YAML served via MCP). The knowledge survives team changes, is reviewable in pull requests, and is available to every AI tool that connects.

## The Encoding Process

You cannot automate institutional knowledge extraction — by definition, it requires the people who have the knowledge. But you can make the process fast and conversational.

ContextKit's `enrich` command, run inside an AI coding assistant like Claude Code or Cursor, creates a structured dialogue. The agent looks at your schema and asks specific questions:

- *"The `amount` column in `transactions` — is this in dollars or cents?"*
- *"Are there any statuses that should always be filtered out?"*
- *"Is `v1_customers` still in active use, or has it been replaced?"*

You answer in plain English. The agent writes the YAML. You review and correct. In an hour, you can encode institutional knowledge that took years to accumulate — not all of it, but enough to prevent the most common and costly errors.

The process is also self-reinforcing. Every time an AI agent writes wrong SQL because of missing context, that error tells you exactly what institutional knowledge to encode next. Add a guardrail, update a description, write a golden query. The semantic layer gets smarter every time it fails.

## Start With What You Know

You do not need to encode every piece of institutional knowledge at once. Start with the knowledge that prevents the most expensive errors:

1. **Required filters.** What rows must always be excluded? Cancelled orders, test accounts, internal users, deprecated records.
2. **Metric definitions.** How are your KPIs calculated? What aggregation, what filters, what time grain?
3. **Table provenance.** Which tables are canonical, which are deprecated, which are staging?
4. **Column semantics.** What do ambiguous columns actually mean? What are the valid values?
