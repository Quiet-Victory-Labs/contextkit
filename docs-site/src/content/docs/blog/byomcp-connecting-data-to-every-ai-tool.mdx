A year ago, connecting your database context to an AI coding assistant meant writing custom prompts, pasting schema snippets into chat windows, or building bespoke integrations for each tool. Today, there is a better way: Model Context Protocol (MCP), an open standard that lets AI tools request structured context from external servers.

BYOMCP — Bring Your Own MCP — is the idea that you should own your data context and serve it to *every* AI tool in your stack from a single source. Not a different integration for Claude, another for Cursor, and another for Copilot. One semantic layer, one MCP server, every tool.

## What Is MCP?

Model Context Protocol is an open protocol, originally developed by Anthropic, that standardizes how AI applications request and receive context from external sources. Think of it as a USB port for AI tools — a standard interface that any tool can plug into.

An MCP server exposes three types of capabilities:

- **Resources** — structured data that an AI tool can read (like your table metadata, column descriptions, or golden queries)
- **Tools** — functions that an AI tool can call (like searching for a table or looking up a metric definition)
- **Prompts** — reusable prompt templates that guide AI behavior for specific tasks

When an AI tool connects to an MCP server, it can discover what resources and tools are available and use them to answer user questions with better context.

The protocol supports two transport modes: **stdio** (for local tools like Claude Code and Cursor) and **SSE/HTTP** (for remote connections). This means the same MCP server can serve a local coding assistant and a remote AI application.

## The BYOMCP Concept

Most data teams today have context scattered across multiple systems — a data catalog here, a dbt project there, some tribal knowledge in Slack threads. When an AI agent needs context about your data, it has nowhere to look.

BYOMCP flips this. Instead of hoping each AI tool has its own data understanding, you build your data context once and serve it everywhere:

1. **You own the metadata.** It lives in your repository as versioned YAML files, not locked inside a vendor.
2. **You curate it once.** Descriptions, semantic roles, guardrails, and golden queries are written once and apply everywhere.
3. **You serve it to every tool.** One MCP server endpoint, consumed by every AI tool that supports the protocol.

The result is that Claude Code, Cursor, Windsurf, GitHub Copilot, and any future MCP-compatible tool all get the same high-quality data context. When you improve a description or add a guardrail, every tool benefits immediately.

## Setting Up BYOMCP with ContextKit

ContextKit makes BYOMCP practical. Here is how to go from zero to serving your data catalog via MCP in about 15 minutes.

### 1. Install ContextKit

```bash
npx create-contextkit
```

Or install globally:

```bash
npm install -g @runcontext/cli
context setup
```

### 2. Build your semantic layer

Connect to your database and introspect the schema:

```bash
context introspect
```

This generates Bronze-tier metadata — table names, column names, types, and relationships. Then enrich it with business context:

```bash
context enrich
```

Working in an AI coding assistant like Claude Code or Cursor, the enrichment process is conversational. The agent proposes descriptions and you confirm or correct them. In an hour, you can reach Silver or Gold tier for your core data products.

### 3. Start the MCP server

```bash
context serve --stdio
```

That is it. Your semantic layer is now available via MCP.

### 4. Connect your AI tools

**Claude Code** — Add to your MCP configuration:

```json
{
  "mcpServers": {
    "contextkit": {
      "command": "context",
      "args": ["serve", "--stdio"]
    }
  }
}
```

**Cursor** — Add the same configuration to Cursor's MCP settings file.

**Windsurf** — Configure in the MCP panel using the same command.

**GitHub Copilot** — MCP support is available in VS Code with the Copilot extension. Add the server configuration in the MCP settings.

Every tool connects to the same server, reads the same metadata, and applies the same guardrails. One source of truth.

## What Your AI Tools Get

Once connected, your AI tools can access:

**Table and column descriptions.** Instead of guessing what `txn_amt` means, the agent reads: "Transaction amount in USD cents for completed purchases."

**Semantic roles.** The agent knows that `revenue` is a measure (aggregate with SUM), `region` is a dimension (group by), and `updated_at` is metadata (filter, don't aggregate).

**Guardrails.** Required filters are enforced automatically. If every revenue query must include `WHERE status = 'completed'`, the agent knows this before it writes a single line of SQL.

**Golden queries.** For common business questions, the agent can reference verified SQL instead of generating from scratch. "What was total revenue last quarter?" has a known-correct answer.

**Relationship metadata.** Join paths, cardinality, and foreign key information prevent the silent row duplication that plagues AI-generated queries.

## Why This Matters Now

The AI tool landscape is moving fast. New MCP-compatible tools launch regularly. Teams adopt and switch between coding assistants based on features, pricing, and preference. In this environment, investing in tool-specific integrations is fragile — the tool you customize for today may not be the tool you use next quarter.

BYOMCP is a hedge against that churn. Your investment is in the metadata — the descriptions, guardrails, and golden queries that encode your team's domain knowledge. That investment is portable. It works with today's tools and tomorrow's tools, because it targets a protocol, not a product.

It also means that different team members can use different AI tools and still get consistent data context. The data engineer using Claude Code and the analyst using Cursor both see the same business definitions and the same guardrails. Consistency comes from the metadata layer, not from standardizing on a single AI tool.

## Local-First, Privacy-Preserving

An important property of the BYOMCP setup: your data context stays local. The MCP server runs on your machine (or your team's infrastructure). Metadata is served directly to the AI tool's context window — it is not uploaded to a third-party service, stored in someone else's database, or used for training.

This matters for teams with sensitive schemas, proprietary business logic, or compliance requirements. You get the benefits of rich AI context without sending your data catalog to an external platform.
