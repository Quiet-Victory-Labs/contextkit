Every data team has metadata. It lives in Alation, Atlan, DataHub, dbt docs, Notion pages, Confluence wikis, and spreadsheets that someone emailed around in 2022. The problem is not the absence of metadata. The problem is that none of it is portable.

When your metadata is locked inside a vendor's proprietary format, you cannot move it, version it, or serve it to the AI tools that increasingly need it. Open Semantic Interchange (OSI) is a specification designed to fix that — a portable, YAML-based metadata standard that works across tools, teams, and AI agents.

## The Lock-In Problem

Most data catalog tools store metadata in their own internal format. This creates three compounding problems:

**You cannot version it.** Metadata changes as your schema evolves, but proprietary catalogs store it in a database, not in files. You cannot diff it, review it in a PR, or roll it back when something breaks. Your metadata has no history.

**You cannot move it.** If you switch catalog vendors — or if you want to use multiple tools — you have to re-create your metadata from scratch. The descriptions, tags, business definitions, and ownership annotations you spent months building are trapped in a system you may eventually outgrow.

**You cannot serve it to AI.** AI coding assistants and autonomous agents need structured, machine-readable metadata at query time. Most catalog tools were designed for human browsing, not machine consumption. They offer web UIs and REST APIs, but not the kind of structured, file-based metadata that AI tools work best with.

The result is that teams invest heavily in metadata curation and then cannot use that investment where it matters most: at the point where AI agents are writing SQL, generating reports, and making data-driven decisions.

## What Is Open Semantic Interchange?

Open Semantic Interchange (OSI) is a YAML-based specification for describing data products and their semantic context. It defines a standard structure for:

- **Table and column metadata** — descriptions, data types, semantic roles, tags
- **Relationships** — foreign keys, join paths, cardinality
- **Guardrails** — required filters, deprecated columns, access notes
- **Golden queries** — verified SQL for common business questions
- **Business context** — metric definitions, ownership, SLA information

An OSI file looks like this:

```yaml
osi: "1.0"
data_product: customer_analytics
source:
  dialect: postgres
  database: analytics
  schema: public
tables:
  - name: customers
    description: "Active customer accounts"
    columns:
      - name: lifetime_value
        type: decimal
        description: "Total spend in USD across all completed orders"
        semantic_role: measure
        aggregation: SUM
        tags: [revenue, kpi]
    guardrails:
      - column: status
        operator: "="
        value: "active"
        reason: "Exclude churned and test accounts from customer metrics"
```

The key design decisions in OSI are intentional:

**YAML, not JSON or XML.** YAML is human-readable and diff-friendly. When metadata changes, the diff is clear and reviewable in any code review tool.

**Files, not databases.** OSI metadata lives in your repository as `.yaml` files. It is versioned with git, reviewed in pull requests, and deployed alongside your code.

**Flat and explicit.** No inheritance hierarchies, no implicit defaults, no magic. Every piece of context is stated directly where it is used. This makes the format easy to parse — for humans and for machines.

**Tiered maturity.** OSI defines three tiers (Bronze, Silver, Gold) that describe how complete and curated the metadata is. This lets teams start with auto-generated metadata and improve incrementally.

## Portability in Practice

The portability of OSI means your metadata works in multiple contexts without conversion:

### Version control

Because OSI is plain YAML files in a git repository, your metadata gets the same workflow as your code. You can see exactly when a description changed, who changed it, and why. You can revert a bad change in seconds. You can branch metadata changes and test them before merging.

### Tool independence

If you start with ContextKit and later adopt a different catalog tool, your OSI files go with you. If you use multiple tools — say, ContextKit for AI serving and a commercial catalog for governance — they can both read the same OSI files. There is no export/import step, no lossy conversion, no re-keying descriptions by hand.

### AI consumption

OSI files are designed to be read by machines. When served via MCP (Model Context Protocol), an AI agent can look up the description, aggregation rule, and guardrails for any column before writing a query. The structured format means the agent does not need to parse natural-language documentation or navigate a web UI — it reads the metadata directly.

### Team collaboration

Because the format is human-readable YAML, domain experts can contribute metadata without learning a specialized tool. A finance analyst can open a YAML file, add a description for a revenue column, and submit a pull request. The barrier to contribution is a text editor and basic git knowledge.

## The Three Tiers

OSI defines maturity tiers that make adoption incremental:

**Bronze** is auto-generated. Run a schema introspection tool and you get Bronze metadata immediately — table names, column names, types, nullability. It is useful for basic AI context but contains no business semantics.

**Silver** adds human-authored context. Every table has a description. Every column has a semantic role (dimension, measure, metadata, identifier). Tags organize columns into business domains. Silver metadata passes lint checks and is meaningfully useful for AI agents.

**Gold** is production-ready. It includes guardrails (required filters that prevent data errors), golden queries (verified SQL for common questions), and complete business context. Gold metadata means an AI agent can answer any common business question about the data product correctly on the first try.

The tier system means you do not need to achieve Gold coverage on day one. Start at Bronze (minutes of work), improve to Silver (an hour or two with an AI assistant), and reach Gold for your most critical data products over time.

## Adoption

OSI is an open specification. ContextKit implements it as the native metadata format, but the spec is available for any tool to adopt. The goal is not to create another proprietary format but to establish a common interchange layer that makes metadata portable across the ecosystem.

If you already have metadata in another system, ContextKit can import and convert it to OSI format. If you are starting from scratch, ContextKit's `introspect` and `enrich` commands build OSI files directly from your database schema.

The specification is versioned (currently `1.0`) and designed for backward compatibility as it evolves.

## Getting Started with OSI

The fastest way to create OSI-compliant metadata is with ContextKit:

```bash

context enrich        # AI-assisted enrichment to Silver
context build         # validates and reports tier status
```

> *"Install @runcontext/cli and build a semantic layer for my database."*

Your metadata is too valuable to lock inside a single vendor. OSI makes it portable, versionable, and ready for the AI-native future of data work.
